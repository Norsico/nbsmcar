# 计算优化索引位置

## 说明

- 本文件只做 `project/` 当前主工程的“可优化计算点筛查”，本轮不改任何业务代码。
- 本轮先看实时主链：图像处理、前轮控制、后轮控制、阿克曼、编码器。
- 目标不是“见到 32 位就砍”，而是先区分：
  - 哪些地方确实在高频路径里，适合先去 `float`、去软件 32 位乘除。
  - 哪些地方虽然有 `int32/uint32`，但直接改成 16 位会有溢出风险，暂时不建议动。

## 首轮结论

- 当前最值得优先看的，不是零碎的 `uint32`，而是高频主链中的 `float`。
- 资源压力最大的嫌疑点主要集中在：
  - `project/code/search_line.c`
  - `project/code/steer.c`
  - `project/code/ackerman.c`
  - `project/code/pid_control.c`
  - `project/code/dev/dev_wheel.c`
- 其中最可疑的几个点：
  - 舵机 10ms 链路里的浮点 PD 和浮点占空比转角度。
  - 后轮 5ms 链路里的浮点 PID 和阿克曼 `tan()`。
  - 图像链里大量“斜率补线 / 前瞻加权 / 直线判定 / 大津法”的浮点运算。

## A级优先

| 级别 | 位置 | 当前情况 | 可优化方向 | 备注 |
| --- | --- | --- | --- | --- |
| A1 | `project/code/steer.c:31-60` | `SteerControl()` 每次舵机控制都用 `float` 做 duty 到 angle 的线性换算。 | 改成定点比例；把分子分母预先放大成整数，运行时只做整数乘除。 | 这是 10ms 链路，频率高。 |
| A1 | `project/code/steer.c:63-90` | `SteerPID_Realize()` 全链路都是 `float`；`0.3f`、`1.2f`、`err2/10.0f`、`P*error`、`D*diff` 都在跑浮点。 | 整体改成定点；例如 `0.3 -> 3/10`，`1.2 -> 12/10`，`err2` 本来就已经是十分位参数，可统一成整数缩放。 | 这块非常贴合“`kp=9, *error / 10`”这种改法。 |
| A1 | `project/user/app/app_line.c:169` | 舵机入口为了调用 `SteerPID_Realize(float)`，先把整数偏差转成了 `float`。 | 如果后续把舵机链改成整型，这里可以直接传 `int16 offset`，把入口的浮点也一并去掉。 | 单点收益不大，但适合作为舵机链收口点。 |
| A1 | `project/code/ackerman.c:43-68` | 阿克曼每次更新都用 `float`，且调用了 `tan()`；这类库函数在单片机上通常比普通四则运算更贵。 | 优先考虑查表或分段表；舵角范围固定，可提前做 `0~30` 度查表，运行时只做整数索引和整数乘法。 | 这是后轮 5ms 链路中的重计算点。 |
| A1 | `project/code/dev/dev_wheel.c:198-209` | `car_wheel_update_reference_target()` 先取 `float` 速度，再取 `float` 舵角，再调用阿克曼，最后还是回到轮速目标。 | 与阿克曼一起整体改成整型链；车速、舵角、左右轮目标都用整数或定点表示，避免这一段来回进出浮点。 | 如果阿克曼不改，这里收益有限。 |
| A1 | `project/code/pid_control.c:34-45` | 后轮增量 PI 用 `float error / float output / float kp / float ki`。 | 用定点 PI；例如 `kp` 放大 10 倍或 100 倍，`ki` 同样放大，输出仍保持 `int16/int32`。 | 后轮 5ms 闭环，高频且持续运行。 |
| A1 | `project/code/search_line.c:1046-1148` | `GetDet()` 里有 `float DetTemp / UnitAll / SpeedGain`，还用了 `Weighting[10]` 浮点权重表。 | 权重表改成整数表，例如 `96, 92, ...`；累加改成 `int32`；最终统一除以缩放因子。 | 前瞻误差是主链核心，调用频率高。 |
| A1 | `project/code/search_line.c:1177-1237` | `Straight_Judge()` 全函数都是浮点斜率、浮点误差平方、浮点方差。 | 改成整数斜率或直接改成整数残差累计；很多地方只是拿它跟阈值比较，不一定非要得到真实浮点方差。 | 该函数在元素判断里会被频繁调用。 |
| A1 | `project/code/search_line.c:740-836` | 白边延长线判断 `D_R / D_L` 使用浮点斜率。 | 用整数斜率或“增量补线”方式替代；优先避免频繁除法。 | 图像主循环内执行。 |
| A1 | `project/code/search_line.c:928-937` | 左边补线 `DetL` 使用浮点斜率，再逐行 `DetL * row_delta`。 | 改成整数增量或 Bresenham 式补线。 | 高频路径。 |
| A1 | `project/code/search_line.c:978-987` | 右边补线 `DetR` 同样使用浮点斜率。 | 同上，改整数增量。 | 高频路径。 |
| A1 | `project/code/search_line.c:1028-1034` | `RouteFilter()` 中线滤波补线再一次用了浮点斜率。 | 同样适合改成整数增量。 | 图像每帧都会走。 |
| A1 | `project/code/search_line.c:1556-1595` | 左环进环补线使用 `Slope_Rings` 浮点斜率。 | 环岛补线逻辑可以统一改成整数斜率模板。 | 只在元素内触发，但一旦触发会连续跑。 |
| A1 | `project/code/search_line.c:1792-1831` | 右环进环补线使用 `Slope_Right_Rings` 浮点斜率。 | 同上。 | 左右环最好一起整理。 |

## B级优先

| 级别 | 位置 | 当前情况 | 可优化方向 | 备注 |

## 暂不建议直接动

| 位置 | 当前情况 | 暂不建议原因 |
| --- | --- | --- |
| `project/code/dev/dev_encoder.c:55-56` | 编码器一阶滤波用了 `int32`：`raw * 8 + last * 92`。 | 这里不是“写得笨”，而是乘法结果本身可能超过 16 位；直接砍成 `int16` 有溢出风险。除非后面把系数整体改写成移位近似，否则先别动。 |
| `project/code/search_line.c:2135-2156` | 局部阈值补偿里用了 `uint32 gray_sum` 和若干 `uint32` 乘除。 | 灰度和、样本数乘法本来就容易超 16 位；这里更适合先确认数值范围，再决定能不能压到 16 位。 |
| `project/code/search_line.c:2591-2631` | 预览绘图坐标缩放用了大量 `uint32`。 | 这是显示链，不是图像控制主链；而且显示节拍较低，优先级明显低于巡线和控制。 |

## 下一轮建议继续看的位置

## 备注

- 本轮是“首轮索引”，先记位置，不下结论说一定要改。
- 真要动代码时，建议先从下面顺序开刀：
  - 舵机链 `steer.c`
  - 后轮链 `ackerman.c + pid_control.c + dev_wheel.c`
  - 图像链 `GetDet() + Straight_Judge()`
  - 最后再看大津法和环岛补线
